Install @clerk/astro and @astrojs/vercel, enable SSR, add Clerk middleware, and wire Convex to validate Clerk JWTs.

Install

NOTE: Clerk requires SSR mode. The Vercel adapter is added here as the deployment target.

src/middleware.ts

Uses clerkMiddleware + createRouteMatcher from '@clerk/astro/server'.

Protected routes: /admin(.*), /dashboard(.*) — calls auth.protect() on match.

convex/auth.config.ts

Configures Convex to validate Clerk JWTs. domain = CLERK_JWT_ISSUER_DOMAIN env var. applicationID = "convex".

After creating/changing this file, run bunx convex dev or bunx convex deploy so the backend picks up the new config.

src/lib/convex.ts — authenticated client helper

Added getAuthedConvex(token) — creates a per-request ConvexHttpClient with auth set.

Usage in Astro SSR: const token = await Astro.locals.auth().getToken({ template: 'convex' }); const client = getAuthedConvex(token);

Required env vars (.env.local)

PUBLIC_CLERK_PUBLISHABLE_KEY=pk_test_...

CLERK_SECRET_KEY=sk_test_...

CLERK_JWT_ISSUER_DOMAIN=https://your-app.clerk.accounts.dev

Activate the Convex integration in Clerk at: https://dashboard.clerk.com/apps/setup/convex

PUBLIC_CLERK_SIGN_IN_URL=/sign-in

PUBLIC_CLERK_SIGN_UP_URL=/sign-up

PUBLIC_CLERK_SIGN_IN_FALLBACK_REDIRECT_URL=/dashboard

PUBLIC_CLERK_SIGN_UP_FALLBACK_REDIRECT_URL=/onboarding

Auth flow pages

src/pages/sign-in.astro — renders <SignIn /> from @clerk/astro/components. Redirects to /dashboard if already signed in.

src/pages/sign-up.astro — renders <SignUp />. Redirects to /dashboard if already signed in.

src/pages/onboarding.astro — POST handler creates Convex user record via api.users.createOrUpdate. Pre-fills name/email from Clerk session. Redirects to /dashboard on success.

Protected pages

src/pages/dashboard/index.astro — requires auth, requires Convex user record (else → /onboarding). Shows user info card.

src/pages/admin/index.astro — requires auth + role === 'admin' (else → /dashboard). Shows admin module nav grid.

convex/users.ts

createOrUpdate mutation — upserts user by clerkId. Called from onboarding POST.

getByClerkId query — looks up user by clerkId. Used on dashboard + admin to gate access.

Middleware gotcha

In @clerk/astro v3, the middleware handler receives auth as a FUNCTION. Call auth() to get the auth object with redirectToSignIn(). auth.protect() does NOT exist — calling it throws "auth.protect is not a function".

Correct pattern: const authObject = auth(); if (!authObject.userId) return authObject.redirectToSignIn();

8. Admin Settings Page

Add a Convex-backed settings page at /admin/settings for application configuration, feature flags, notifications, design system brand colors, and a danger zone.

convex/schema.ts — appSettings table (singleton)

Added appSettings table — no index needed (singleton pattern, queried with .first()).

Fields: appName, siteUrl, supportEmail, description (general); maintenanceMode, registrationEnabled, blogEnabled (feature flags); adminAlertEmail, notifyOnContact, notifyOnNewUser (notifications); primaryColor/Name, secondaryColor/Name, tertiaryColor/Name, neutralColor/Name (design system).

All fields are v.optional() — first insert creates the doc, subsequent saves use ctx.db.patch() to merge only the changed section.

convex/settings.ts

get (public query) — returns .first() from appSettings, or null if unconfigured.

upsert (mutation) — patches existing doc or inserts new. All args optional. Called per-section from the settings page POST handler.

reset (mutation) — deletes the appSettings document, reverting to defaults.

src/components/DesignSystem.astro — props added

Accepts optional props: primary, secondary, tertiary, neutral — each { name: string; hex: string }.

When a custom hex is provided, uses inline style="background-color: {hex}" on the swatch card instead of the Tailwind bg class. Falls back to hardcoded defaults when no props are passed.

src/pages/index.astro — pass design settings to DesignSystem

Fetches settings via the public convex client (no auth required). Passes stored color overrides as props to <DesignSystem />.

src/pages/admin/settings.astro

Auth-gated (admin only). Handles POST with section routing: general, features, notifications, design, danger.

5 sections: General (app name, site URL, support email, description), Feature Flags (3 toggles), Notifications (admin alert email + 2 event toggles), Design System (4 brand colors with color pickers + live swatch previews), Danger Zone (reset settings to defaults).

Inline <script> syncs color picker → hex text display → swatch preview on input.

Toggle switches use CSS-only peer pattern (sr-only checkbox + peer-checked: Tailwind classes).

9. Admin Pages Management

Add a Convex-backed admin page at /admin/pages to control public page visibility. Status overrides are stored in Convex; the canonical page catalog lives in a shared TypeScript file.

src/lib/pages.ts — shared page catalog

Exports PAGE_CATALOG (array of 8 pages with name, route, description, category, defaultStatus, colorClasses) and CATEGORY_COLORS map.

This is the single source of truth for the page list. Both SitePages.astro and admin/pages.astro import from here — no duplication.

convex/schema.ts — sitePages table

Table: sitePages — fields: route (string), status (union: 'active' | 'planned' | 'hidden').

Index: by_route. Only pages with non-default statuses need a record — catalog defaults to 'planned' for missing routes.

convex/sitePages.ts

listStatuses (public query) — returns all sitePages records.

setStatus (mutation) — upserts a route's status using by_route index + .unique(). Called from admin POST handler.

src/components/SitePages.astro — updated

Now fetches Convex status overrides via public client. Merges with PAGE_CATALOG defaults. Displays active / hidden / planned status badges.

No longer has hardcoded page data — all from the shared catalog.

src/pages/admin/pages.astro

Auth-gated (admin only). POST handler validates route and status, calls api.sitePages.setStatus.

Table view: page name/icon, route (monospace badge), category chip, current status badge, contextual action buttons (Activate / Hide / Mark planned — only shows buttons that change the current state).

Stats summary (active / hidden / planned counts) in the header. Status legend below the table.

10. Dynamic Admin Sections — Funnels, Shops, Booking Links

Three user-created entity types with identical data shape. Each gets a Convex table, a list page, and a per-instance edit page. The admin portal index fetches all three at render time and shows one card per instance plus a "Create new" dashed card at the end of each section.

convex/schema.ts — three new tables

funnels, shops, bookingLinks — identical shape: name (string), slug (string), description (optional string), published (boolean).

Indexes: by_slug and by_published on each table.

convex/funnels.ts, shops.ts, bookingLinks.ts

Each exports: list (query, ordered desc, take 100), get (query by id), create (mutation), update (mutation — patches by id), remove (mutation).

List pages (index.astro per entity)

src/pages/admin/funnels/index.astro, /shops/index.astro, /booking-links/index.astro.

Pattern: POST handler with action=create, collapsible create form (toggled via inline JS), table of existing records linking to [id] edit pages. Empty state with inline prompt.

Slug is auto-generated from the name input via a client-side script (tracks if user manually edits slug to stop auto-fill).

Edit pages ([id].astro per entity)

src/pages/admin/funnels/[id].astro, /shops/[id].astro, /booking-links/[id].astro.

POST with action=update patches the record. action=delete calls remove mutation then redirects to list. 404-guards via redirect if record not found. Danger zone section with confirm dialog.

src/pages/admin/index.astro — dynamic sections

Fetches all three lists in parallel via Promise.all at the top of the frontmatter.

Static sections (Management, Content) remain unchanged. Dynamic sections (Funnels, Shops, Booking Links) rendered below: one card per DB record linking to /admin/{entity}/{id}, plus a dashed "+ New" card linking to the list page. Each section has a "View all →" link.

12. Admin Communications Module

Full email management hub in the admin portal: compose one-off emails, broadcast to all users, manage reusable templates, and view a send history log. Powered by Resend via two new Convex actions.

convex/schema.ts — two new tables

emailTemplates — fields: name, subject, htmlBody. Index: by_name.

emailLogs — fields: type (single | broadcast), recipients (array), subject, templateId (optional), sentByClerkId, recipientCount. No index needed — queried desc by _creationTime.

convex/emailTemplates.ts

list (query), get (query by id), create (mutation), update (mutation), remove (mutation).

convex/emailLogs.ts

create (internalMutation) — only called from email actions. list (query, order desc, take 100).

convex/email.ts — new public actions

sendEmail (action) — auth-gated, admin-only. Sends to single recipient, logs to emailLogs.

sendBroadcast (action) — auth-gated, admin-only. Fetches all users, uses resend.batch.send(), logs to emailLogs.

Both actions: verify identity + admin role via ctx.auth.getUserIdentity() + ctx.runQuery(api.users.getByClerkId). Generate plain-text fallback via stripHtml().

Admin pages (6 files)

src/pages/admin/communications/index.astro — hub with stats (templates, emails sent, total recipients) and 4 feature cards.

src/pages/admin/communications/compose.astro — POST calls client.action(api.email.sendEmail). Users datalist for recipient autocomplete. Template picker pre-fills subject + body via inline script.

src/pages/admin/communications/broadcast.astro — POST calls client.action(api.email.sendBroadcast). Shows user count warning. Confirm dialog before submit.

src/pages/admin/communications/templates/index.astro — list + collapsible create form (POST action=create).

src/pages/admin/communications/templates/[id].astro — edit form (action=update) + danger zone delete (action=delete → redirect).

src/pages/admin/communications/history.astro — read-only log table: type badge, subject, recipients, count, sent timestamp.

src/pages/admin/index.astro — Communications section

Fetches emailTemplates.list and emailLogs.list for live counts. Renders 4 active cards above the dynamic entity sections.

11. Resend Email

Install resend, create an internal Convex email action, wire it to contact form submissions and new user registrations via the admin notification settings.

Install

bun add resend

convex/email.ts — internalAction sendAdminAlert

"use node" — required because Resend SDK may use Node.js built-ins.

Args: to (string), subject (string), html (string).

Reads RESEND_API_KEY and RESEND_FROM_EMAIL from process.env. Returns null silently if API key is missing.

Called via ctx.scheduler.runAfter(0, internal.email.sendAdminAlert, { ... }) from mutations — never called directly from queries.

convex/contacts.ts — CRUD + email trigger

create mutation — inserts contact, then checks appSettings.notifyOnContact + appSettings.adminAlertEmail. Schedules sendAdminAlert if both are truthy.

list (query, desc, take 100), listUnread (query, by_read index), markRead (mutation), remove (mutation).

convex/users.ts — new user email trigger

createOrUpdate mutation updated: on new insert (not update), checks appSettings.notifyOnNewUser + appSettings.adminAlertEmail. Schedules sendAdminAlert if both are truthy.

Required Convex env vars (set via CLI)

bunx convex env set RESEND_API_KEY re_...

bunx convex env set RESEND_FROM_EMAIL noreply@yourdomain.com

NOTE: RESEND_FROM_EMAIL must be a verified domain in Resend. For testing, Resend provides onboarding@resend.dev but only sends to the account owner's email.

Activation via admin settings

Go to /admin/settings → Notifications section. Set adminAlertEmail and toggle on "New contact submission" and/or "New user registration".

No emails send until both the env vars are set in Convex AND the toggles are enabled in settings.

13. PostHog Analytics

Install posthog-js and posthog-node. Add a client-side snippet to BaseLayout and a server-side singleton for tracking events from Astro SSR pages and API routes.

Install

bun add posthog-js posthog-node

.env — four required variables

PUBLIC_POSTHOG_PROJECT_TOKEN=phc_... (exposed to browser via PUBLIC_ prefix)

PUBLIC_POSTHOG_HOST=https://us.i.posthog.com

POSTHOG_PROJECT_TOKEN=phc_... (server-only, no PUBLIC_ prefix)

POSTHOG_HOST=https://us.i.posthog.com

src/components/posthog.astro — client snippet

Uses is:inline + define:vars to inject token and host from env vars without Astro processing the script.

Calls posthog.init() with defaults: '2026-01-30'.

IMPORTANT: Always use is:inline on PostHog script tags. Without it Astro processes the script and throws TypeScript errors.

src/layouts/BaseLayout.astro — include PostHog

import PostHog from '@components/posthog.astro';

Add <PostHog /> inside <head> just before the closing </head> tag.

Because BaseLayout wraps every page, PostHog loads on every route automatically.

src/lib/posthog-server.ts — server singleton

Exports getPostHogServer() — creates one PostHog (posthog-node) instance and reuses it. Reads POSTHOG_PROJECT_TOKEN and POSTHOG_HOST from import.meta.env.

flushAt: 1, flushInterval: 0 for immediate flush in SSR context.

Exports shutdownPostHog() for graceful drain on server shutdown.

Events instrumented (9 events across 7 files)

onboarding_viewed (client) + onboarding_completed (server) — src/pages/onboarding.astro. Server also calls posthog.identify() with name and email.

funnel_created (server) — src/pages/admin/funnels/index.astro

shop_created (server) — src/pages/admin/shops/index.astro

booking_link_created (server) — src/pages/admin/booking-links/index.astro

email_sent (server) — src/pages/admin/communications/compose.astro

email_broadcast_sent (server) — src/pages/admin/communications/broadcast.astro

user_role_changed + user_removed (server) — src/pages/admin/users.astro

User identification pattern

Server: posthog.identify({ distinctId: clerkUserId, properties: { name, email } }) called on onboarding_completed.

Client: window.posthog?.identify(clerkUserId) called on the onboarding page via define:vars passing the Clerk userId from the frontmatter.

All server captures use the Clerk userId (from Astro.locals.auth().userId) as the distinctId for consistent identity.
