astro.config.mjs — Vite watch ignore (required)

Add to vite config: server: { watch: { ignored: ['**/convex/_generated/**'] } }

WHY: bunx convex dev regenerates convex/_generated/ on every schema/function change. Without this, Vite detects those writes as source changes and triggers a full program reload — causing a race condition where astro:server-app.js virtual module is not yet registered, producing "Failed to load url astro:server-app.js" errors in the dev console.

Middleware gotcha

In @clerk/astro v3, the middleware handler receives auth as a FUNCTION. Call auth() to get the auth object with redirectToSignIn(). auth.protect() does NOT exist — calling it throws "auth.protect is not a function".

Correct pattern: const authObject = auth(); if (!authObject.userId) return authObject.redirectToSignIn();

Upload flow — fully server-side (no browser JS required for upload)

1. Form submits as multipart/form-data with action=upload to the Astro SSR POST handler.

2. Server reads the File from formData.get('file') and determines mediaType from file.type (image/* → image, video/* → video, application/pdf → pdf).

3. Server calls client.mutation(api.gallery.generateUploadUrl, {}) to get a short-lived Convex upload URL.

4. Server POSTs the file to that URL: fetch(uploadUrl, { method: 'POST', headers: { 'Content-Type': mimeType }, body: file }).

5. Convex returns { storageId }. Server calls api.gallery.create with storageId + metadata.

NOTE: Client-side drag-and-drop enhancement is present (updates the filename label and injects dropped files into the input), but the actual upload always goes through the SSR POST handler.

Step 30 — Email Unsubscribe / Opt-Out Mechanism (CAN-SPAM + GDPR Compliance)

What was added

All three agent email sequences (Agent 3 Reactivation, Agent 4 Reviews & Referrals, Agent 5 Lead Nurturing) previously sent emails with no unsubscribe mechanism. CAN-SPAM and GDPR require a clear and functional opt-out in every marketing/promotional email. This step wires a fully tokenized unsubscribe flow end-to-end.

Schema: optedOut: v.optional(v.boolean()) added to the contacts table in convex/schema.ts. Convex handles the migration automatically — no manual migration step needed.

Mutation: contacts.optOut (public mutation) patches optedOut: true on the contact record. Takes a single arg: contactId: v.id('contacts').

Token signing: The unsubscribe link uses the same HMAC-SHA256 pattern as booking cancellation and reactivation YES/NO. Token payload: unsub:${contactId}. Signed with CANCEL_SECRET (must be identical in both Vercel env vars and Convex env vars). Token generation happens in each *Actions.ts via Node.js crypto.createHmac. Token verification happens in src/lib/tokens.ts via verifyToken(payload, token).

Unsubscribe footer HTML: A unsubFooterHtml(contactId) helper function was added to nurturingActions.ts, reactivationActions.ts, and reviewActions.ts. It generates: ${siteUrl}/api/unsubscribe?contactId=${contactId}&token=${token}. If CONVEX_SITE_URL is not set, the function returns an empty string — the email sends without a footer rather than breaking. The footer is appended to every outbound email HTML body via the existing extraHtml parameter of emailHtml().

Opt-out guard: At the top of each action handler (before template resolution), the code checks contact.optedOut. If true, it logs a skipped entry with errorMessage: "opted_out" and returns early — no email is sent. For booking-based actions (nurturing M4–M5, reviews R1–R4), there is no direct contactId, so the action looks up the contact by email using the new internal.nurturing.getContactByEmail internal query. If no contact record is found, the opt-out check is skipped (graceful) and the footer link is omitted.
