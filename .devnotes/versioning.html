Versioning Policy

c4studio B.A.S. template — how we number releases and what each increment means

1. What is Semantic Versioning?

This project follows Semantic Versioning 2.0.0 (semver.org/spec/v2.0.0.html). Every release gets a three-part version number in the exact form X.Y.Z:

Part Name Increment when…
X MAJOR You make a breaking, backwards-incompatible change to the public API or template contract. Client projects forked from a previous major version may need manual migration work.
Y MINOR You add new functionality in a backwards-compatible way. A client on v1.2.0 can safely upgrade to v1.5.0 without breaking anything.
Z PATCH You ship backwards-compatible bug fixes only. No new features, no breaking changes — purely corrective.
Format rules (non-negotiable)
  • All three parts are required: 1.0 is invalid — 1.0.0 is correct.
  • No leading zeroes: 01.2.3, 1.02.3, 1.2.03 are all invalid.
  • No spaces: v 1.2.3 is invalid — v1.2.3 is correct.
  • The v prefix is a convention (not part of the spec) — use it consistently.

2. The 11 Core Rules (semver 2.0.0)

1

Declare a public API.

Software using semver MUST declare a public API — in code, documentation, or both. This project's "API" is the template contract: file structure, path aliases, Convex schema conventions, admin auth guard pattern, and agent module pattern.
2

Version format is X.Y.Z with non-negative integers and no leading zeroes.

Each element MUST increase numerically. Example: 1.9.0 → 1.10.0 → 1.11.0 (not 1.9.0 → 1.9.1 if you added a feature).
3

Once released, a version is immutable.

You MUST NOT modify the contents of a released version. If something is wrong, release a new version. Retroactively editing what v0.0.6 means breaks every project that pinned to it.
4

Major version zero (0.y.z) is for initial development.

Anything may change at any time during 0.x. The public API should not be considered stable. This is where B.A.S. lives until v1.0.0.
5

Version 1.0.0 defines the first stable public API.

For B.A.S., this is the Production Ready milestone — all blockers resolved, env var verification done, build signs off cleanly.
6

Patch version Z MUST be incremented for backwards-compatible bug fixes only.

A bug fix is an internal correction of incorrect behaviour. Example: fixing the design system color picker sync is a patch — it corrects broken behaviour without adding anything new.
7

Minor version Y MUST be incremented when new, backwards-compatible functionality is added.

When Y increments, Z resets to 0. Example: adding SMS delivery (v1.1.0) is additive — existing client projects are unaffected but gain the capability if they pull in the update.
8

Major version X MUST be incremented for any backwards-incompatible changes.

When X increments, both Y and Z reset to 0. Example: the multi-tenant architecture rewrite (v2.0.0) will require client projects to migrate.
9

Pre-release versions use a hyphen suffix.

Examples: 1.0.0-alpha, 1.0.0-alpha.1, 1.0.0-rc.1. A pre-release version has lower precedence than the normal version: 1.0.0-rc.1 < 1.0.0.
10

Build metadata uses a plus-sign suffix.

Examples: 1.0.0+20240610, 1.0.0-beta+exp.sha.5114f85. Build metadata does NOT affect version precedence and is generally not used in this project.
11

Precedence is determined by comparing major, minor, patch numerically.

Pre-release identifiers are compared field by field left to right. Full chain: 1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta < 1.0.0-rc.1 < 1.0.0.

3. What counts as MAJOR / MINOR / PATCH for B.A.S.

Change typeIncrementExamples
Breaking changes to the template contract MAJOR Renaming path aliases, restructuring convex/schema.ts in a way that requires migration, changing the admin auth guard pattern, switching core stack technologies
New modules / new admin pages / new Convex tables MINOR Adding SMS delivery, adding live agent chat, adding a new commerce module, adding new feature flags, new public pages
Bug fixes to existing functionality PATCH Fixing the color picker sync bug, fixing a booking conflict check edge case, correcting a type error, fixing a broken redirect
Pre-release work in progress PRE-RELEASE v1.0.0-rc.1 — a release candidate for production readiness; v0.0.8-alpha — early access to a planned patch

Rule 4 in practice We are in 0.0.z — the public API is not yet stable. Anything may change between patch versions during this phase. Once v1.0.0 ships, the stable API contract applies: breaking changes require a major bump, new features require a minor bump, and fixes are patches only.

5. Reset Rules

When you bump a higher-order version, lower-order versions reset to zero:

1.4.3 +patch → 1.4.4
1.4.3 +minor → 1.5.0 (patch resets to 0)
1.4.3 +major → 2.0.0 (minor and patch reset to 0)

6. Pre-release Identifiers

Append a hyphen and dot-separated identifiers after the patch number:

IdentifierMeaningExample
alphaEarly, unstable. Internal only. Breaking changes expected.v1.0.0-alpha
betaFeature-complete but not yet hardened. Limited external testing.v1.0.0-beta.1
rc.NRelease candidate. Considered stable unless a showstopper is found.v1.0.0-rc.1

Precedence (lower → higher):

1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-beta < 1.0.0-rc.1 < 1.0.0

8. Where Versions Appear in This Repo

  • src/pages/changelog.astro — authoritative shipped history; add a new entry for every release
  • src/pages/roadmap.astro — planned and in-progress milestones; version strings must be valid semver
  • README.md — canonical [x]/[ ] task checklist; update when tasks complete
  • CLAUDE.md — architecture docs; update Outstanding Work section when milestones ship
  • .devnotes/project-setup.html — human/agent handoff doc; must be updated on every change
Keep changelog and roadmap in sync When a roadmap milestone ships, move it from the roadmap to the changelog with the same version number. Do not reuse or renumber versions — the changelog entry becomes the immutable record of what that version delivered.
